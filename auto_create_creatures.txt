Gegner-Statblock Spezifikation v1.1
Zweck
Einheitliche Regeln und Formate, damit eine AI spielbare Gegner-Statblöcke erzeugen kann, die ohne Rückfragen am Tisch funktionieren.
Globale Regeln (Explizitheit)
Jede verkürzte/keywordartige Angabe in irgendeinem Feld muss unter Besonderheiten kurz und eindeutig erläutert werden (Wirkung, Grenzen, Beispiele).
Beispiele: immun, Resistenz, Anfälligkeit, „0 vs …“, „halber/doppelter Schaden“, „körperlos“, „Regeneration“, „Vorteil/Nachteil“, Sonderkrits („kritisch bei 19-20“), „automatisch“ etc.
Wenn z.B. RW: immun steht, MUSS unter Besonderheiten eine Klartext-Zeile wie „Immun gegen alle geistesbeeinflussenden Effekte (z.B. Furcht, Einschüchtern, Schlaf)“ stehen.
Top-Level-Felder (Pflicht, falls nicht anders markiert)
name: String
typ: String (z.B. Humanoid, Untot, Tier, Monster, Riese, Elementar, Schleimwesen, Gestaltwandler)
hg: Integer (1,2,…) oder Bruch als String („1/8“, „1/4“, „1/2“)
ep: Integer (Richtwerte siehe „Designziele“, systemabhängig anpassbar)
lp: Integer
sr: Integer
sr_bemerkung: Optionaler String (z.B. „Lederfetzen“, „0 vs Wuchtwaffen“)
bew: String. Format:
Nur Boden: „4“
Mit weiteren Modi: „4 (Flug 7, Schwimmen 3)“
pa: Integer
asw: Integer
ini: Integer (darf negativ sein)
rz: Integer oder „immun“
rr: Integer oder „immun“
rw: Integer oder „immun“
pa_bemerkung, asw_bemerkung: Optional String
angriffe: Array von Angriffsobjekten (siehe unten)
besonderheiten: Array von Strings (freier Text; Explizitheitspflicht beachten)
beute: Array von Beuteobjekten (siehe unten)
beschreibung: String (Pflicht). Kurzer, regelarmer Flavour-Text gemäß Abschnitt „Beschreibung – Anforderungen“.
Angriffe (Objektstruktur im Array „angriffe“) Pflicht
name: String (z.B. „Kurzschwert“, „Biss“, „Schallschrei“)
to_hit: Integer (Angriffsbonus; nicht „bonus“)
schaden: Zahl oder Würfelstring (z.B. 5 oder „1W6+2“)
Optionale Felder (nutzen, wenn relevant)
reichweite: String-Token
Erlaubte Tokens: „angrenzend“, „1xKurz“, „2xLang“, „3xLang“, „Kegel 3“, „Linie 6“
anzahl: Integer (Einsätze pro Begegnung, z.B. 1 für 1×)
zusatz: Kurztext (z.B. „Ziel zu Boden“, „festgesetzt“)
rettungswurf: Objekt, Pflicht wenn der Angriff einen Save/Zustand/Effekt auslöst
art: „RZ“ | „RR“ | „RW“
zw: Integer (Zielwert)
bei_misserfolg: Kurztext der Wirkung (z.B. „1 Runde betäubt“, „Gift: 1W4/Runde (max 3 Runden)“)
bei_erfolg: Kurztext (z.B. „keine Wirkung“, „halber Schaden“)
dauer: String (z.B. „1 Runde“, „1W4 Runden“, „bis Ende nächste Runde“)
wiederholung: Optional (z.B. „Ende jeder eigenen Runde“, „täglich“)
tags: Optional Array (z.B. [„gift“, „krankheit“, „flächenangriff“])

Hinweise
Jeder Fern-/Flächenangriff braucht eine reichweite.
Jeder Effekt/Zustand braucht ein rettungswurf-Objekt (inkl. Erfolg/Misserfolg und Dauer).
Wenn ein Angriff per Keyword andeutet (z.B. „Gift“, „Krankheit“), muss die Wirkung klar (rettungswurf) definiert sein.
Zauber (optional)
Format wie Angriffe; zusätzlich erlaubt:
auto: true (falls ohne to_hit)
weitere wirkungsbezogene Felder analog zu reichweite/rettungswurf
Beute (Array) Erlaubte Formen
Gold: { "typ": "GM", "wurf": "2W6" }
Gegenstand immer: { "typ": "Wolfspelz", "wert": 10 } oder { "typ": "Pfeile", "anzahl": 10 }
Prozentdrop: { "wurf": "25% (W20|5)", "typ": "Werkzeuge" }
Prozentdrop mit Auswahl: { "wurf": "20% (W20|4)", "wahl": [ { "typ": "Kurzschwert" }, { "typ": "Pfeile", "anzahl": 10 } ] }
Regeln
Wahrscheinlichkeiten stets als „X% (W20|Y)“ in wurf.
Bei wahl sind Alternativen gleichgewichtet (falls nicht anders angegeben).
anzahl bei Beute bezeichnet Stack/Anzahl des Items (nicht Nutzungsanzahl).
Besonderheiten (Array freier Text)
Kurz, eindeutig. Resistenzen/Immunitäten/Anfälligkeiten hier aufführen.
Globale Explizitheitsregel: Alle Keywords/Abkürzungen aus irgendeinem Feld werden hier in Klartext erläutert.
Bewegung (bew)
Immer Bodenbewegung als String.
Weitere Modi in Klammern: „X (Flug Y, Schwimmen Z)“.
Beschreibung - Anforderungen (normativ): 
- Pflicht: MUSS vorhanden sein.
- Umfang: MUSS 1–3 Sätze umfassen und ca. 25–70 Wörter enthalten.
- Inhalt (Soll-Kriterien): Muss mindestens zwei der folgenden drei Aspekte enthalten:
    1. Erscheinung/Atmosphäre (markanter Eindruck: Aussehen, Geräusch, Geruch).
    2. Verhalten/Taktik (typisches Vorgehen, Jagd-/Kampfstil).
    3. Umfeld/Hook (typische Orte, Begegnungsanlass, kleiner Aufhänger).
- Stilregeln:
    - MUSS regelfrei sein: KEINE Zahlen, Boni, Würfelnotation, Zustandsnamen oder Mechanikbegriffe.
    - MUSS knapp und spielleitungsfreundlich sein (keine Wiederholung von Regeln aus Besonderheiten).
    - SOLL in knapper, bildhafter Prosa geschrieben sein (3. Person, Präsens oder zeitlos).
-  Rollenspezifische Akzente (gemäß Addendum)
    - Bruiser/Schütze/Skirmisher: Fokus auf Bewegung, Positionierung, körperlichen Eindruck.
    - Controller/Unterstützer: Gefühl des Signatureffekts andeuten (ohne Crunch).
- Akzeptankriterien (Validierung):
    - [MUSS] 25–70 Wörter, 1–3 Sätze.
    - [MUSS] Mindestens zwei der drei Inhalte (Erscheinung, Verhalten, Umfeld).
    - [MUSS] Keine Zahlen/Würfel/Regelbegriffe.
    - [SOLL] Keine Dopplung von Besonderheiten-Inhalten; eher Stimmung als Regeln.
Notation und Tokens
Würfel: „XWY+Z“ (z.B. „1W6“, „2W4+1“)
Reichweite: „angrenzend“, „1xKurz“, „2xLang“, „3xLang“, „Kegel N“, „Linie N“
Zustände (Kurzbegriffe, systemneutral formulieren): betäubt, gelähmt, festgesetzt, zu Boden, verlangsamt, in Flammen, furchtsam
Designziele nach HG (Richtwerte)
Diese Werte steuern die Generierung; sie sind kein Teil des Statblocks.
Angriffsbonus in der Tabelle ist ein allgemeiner Richtwert; einzelne Angriffe dürfen ±1 abweichen.
„Schaden/Runde“ nur für internes Balancing; ohne belastbare SC-Referenzen optional.

Generierungsheuristik (für die AI)
Wähle 1–3 Angriffe mit klaren Rollen (Hauptschaden, Kontrolle/Zustand, Reichweite).
Setze to_hit pro Angriff innerhalb des HG-Richtwerts (±1 Abweichung ok).
Gib jedem Effekt vollständige Save-Infos (art, zw, Erfolg/Misserfolg, Dauer; ggf. wiederholung).
Setze reichweite bei allen Nicht-Nahkampfangriffen/Flächeneffekten.
Setze anzahl bei limitierten Aktionen (z.B. 1×).
bew immer als Bodenbewegung angeben; weitere Modi in Klammern.
Beute: Immer GM-Wurf plus 0–2 Prozentdrops. Wahrscheinlichkeiten als „X% (W20|Y)“; „oder“ via wahl.
Prüfe globale Explizitheit: Existieren Keywords in Feldern? Dann passende Klartext-Zeile in Besonderheiten hinzufügen.

Validierungs-Checkliste (Pflichtprüfung)
Top-Level-Felder vorhanden: name, typ, hg, ep, lp, sr, bew, pa, asw, ini, rz, rr, rw, angriffe[], besonderheiten[], beute[].
bew-Format korrekt („X“ oder „X (Flug Y, …)“).
Jeder Fern-/Flächenangriff hat reichweite.
Jeder Effekt/Zustand hat rettungswurf mit art, zw, Erfolg/Misserfolg und Dauer.
anzahl gesetzt, wenn begrenzte Nutzung.
Beute-Wahrscheinlichkeiten im Format „X% (W20|Y)“.
Werte grob im HG-Korridor (LP, SR, to_hit, Verteidigung ≈ PA/ASW).
Globale Explizitheit erfüllt: Für jedes Keyword gibt es eine erklärende Besonderheit.

Option: Interner Damage-Check (nur für Balancing, kein Feld)
Grobformel (falls gewünscht): p ≈ clamp((21 + to_hit – Zielverteidigung)/20, 0.05, 0.95); Erwartungsschaden ≈ p × durchschnittlicher Einzelschaden (Multiattacken/On-Hit/Additionen berücksichtigen). Ohne verlässliche SC-Referenzen optional/überspringbar.


Addendum v1.1a: Vielfalt- und Einfachheitsregeln
Rollenverteilung pro HG (Richtwerte; genau eine Rolle pro Gegner wählen)
HG 1/8–1: Bruiser 45%, Skirmisher 25%, Schütze 15%, Controller 10%, Unterstützer 5%
HG 2–3: Bruiser 35%, Skirmisher 20%, Schütze 15%, Controller 20%, Unterstützer 10%
HG 4+: Bruiser 30%, Skirmisher 20%, Schütze 15%, Controller 25%, Unterstützer 10%
Rollendefinitionen (kurz)
Bruiser: 1–2 Nahkampfangriffe, keine Save-/Zustandseffekte.
Skirmisher: 1 Nahkampf + 1 Fern oder Mobilitäts-Trick; höchstens 1 limitierte Utility (ohne Save).
Schütze: Fokus Fernkampf; optional 1 Spezialschuss, aber ohne Zustandseffekt oder max. 1 Save-Effekt (anzahl 1).
Controller: 1 Save-basierter CC/Flächenangriff (anzahl 1–2), restliche Angriffe simpel.
Unterstützer: Buff/Debuff/Heilung (anzahl), kein zusätzlicher CC-Spam.
Effekt-Budget pro Gegner
Nicht-Controller/Unterstützer: 0 Save-/Zustands-Effekte (Standard). In höchstens 30% dieser Fälle ist 1 kleiner Effekt erlaubt (z.B. „zu Boden“, anzahl 1).
Controller/Unterstützer: max. 1 Save-basierter Effekt (selten 2, wenn beide anzahl-begrenzt).
Begegnungsweite Begrenzung
In einer Begegnung sollen höchstens 1/3 der Gegner Save-/CC-Effekte haben.
Max. 1 AoE-CC pro Begegnung als Standard (Bosskämpfe ausgenommen).
Angriffszahl und Limitierungen
1–2 Angriffe total sind Standard. 3 nur bei Elite/Boss.
Jeder Effektangriff hat anzahl (z.B. 1 oder 2). Dauer/ZW klar angeben (Spec v1.1).
Beispiel-Heuristik für die AI (implizit)
Erzeuge zuerst die Rolle nach obiger Gewichtung, dann:
Bruiser/Skirmisher/Schütze → erstelle nur simple Angriffe; keine CC, außer seltene Ausnahme (anzahl 1).
Controller/Unterstützer → genau 1 Effekt definieren (mit rettungswurf), rest simple.
Kompakter Prompt-Zusatz für deine Generatoren Füge das deiner Promptvorlage an, damit „keine Effekte“ endlich der Normalfall wird:
text


Nutze die Rollenverteilung aus Addendum v1.1a. Wähle genau eine Rolle je Gegner.
- Wenn Rolle ≠ Controller/Unterstützer: Erzeuge KEINE Save-/Zustands-Effekte. Erlaube in höchstens 30% der Fälle EINEN kleinen Effekt (z.B. „zu Boden“), dann mit anzahl=1 und vollständigem rettungswurf.
- Wenn Rolle = Controller/Unterstützer: Erzeuge GENAU EINEN Save-/Zustands- oder AoE-Effekt (anzahl 1–2). Alle übrigen Angriffe sind simpel.
- In Summe: Max. 1/3 der Gegner mit Effekten; höchstens ein AoE-CC pro Begegnung.
